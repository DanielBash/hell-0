# Руководство по внесению вклада (Contributing)

Спасибо, что решили помочь проекту! Ниже описаны основные правила работы с репозиторием, структурой кода и процессами CI/CD.

## 1. Работа с GitHub

Все файлы, связанные с GitHub Actions и оформлением репозитория, находятся в папке **`.github/`**:
- Баннеры и изображения
- CI-процессы, запускаемые при коммите

### Автоматизация при коммите в ветку `main`

При каждом пуше в `main` выполняются следующие шаги:

1. **Запуск тестов** — все тесты из папки `tests/`.
2. **Сборка Docker-образа** — обновление образа на основе `Dockerfile`.
3. **Релиз и обновление версии** — если сообщение коммита начинается с одного из тегов:
   - `[PATCH]` — исправление ошибок (обратная совместимость)
   - `[MINOR]` — новая функциональность (обратная совместимость)
   - `[MAJOR]` — крупные изменения (без обратной совместимости)

> **Важно:** Версионирование следует правилам [Semantic Versioning](https://semver.org/).  
> Теги версий ставьте только при значимых изменениях.

### Рекомендации по ветвлению

Из-за небольшого размера команды строгий Git Flow не обязателен, но **рекомендуется**:
- Разрабатывать новые фичи или исправления в **отдельных ветках** (например, `feature/xxx`, `fix/yyy`).
- Создавать Pull Request для слияния в `main` — это позволяет прогнать тесты и получить ревью.

## 2. Структура проекта

Проект организован следующим образом (корневая директория):

| Файл/папка | Назначение |
|------------|-------------|
| `main.py` | Точка входа в приложение (Flask-приложение) |
| `settings.py` | Основные настройки приложения |
| `.env` | Переменные окружения (не в репозитории, создаётся локально) |
| `requirements.txt` | Зависимости Python |
| `Dockerfile` | Инструкция для сборки Docker-образа |
| `gunicorn_config.py` | Конфигурация Gunicorn для продакшена |
| `.gitignore` | Список игнорируемых Git файлов |

### Основные директории

- **`.github/`** — CI/CD workflows, баннеры, настройки GitHub.
- **`blueprints/`** — модули Flask (регистрация маршрутов).
- **`core/`** — бизнес-логика, вспомогательные модули, работа с данными.
- **`templates/`** — Jinja2-шаблоны для рендеринга HTML.
- **`static/`** — статические файлы (CSS, JS, изображения).
- **`tests/`** — модульные и интеграционные тесты.
- **`settings_templates/`** — примеры файлов настроек (для новых разработчиков).

## 3. Запуск и разработка

1. Клонируйте репозиторий.
2. Создайте виртуальное окружение: `python -m venv venv`
3. Активируйте его и установите зависимости: `pip install -r requirements.txt`
4. При необходимости скопируйте `settings_templates/.env.example` в `.env` и отредактируйте.
5. Запустите приложение: `python main.py`

Для запуска тестов: `pytest tests/`

## 4. Правила оформления коммитов

- Сообщение коммита должно быть осмысленным, на русском языке.
- Если коммит в `main` содержит в начале тег (`[PATCH]`, `[MINOR]`, `[MAJOR]`), будет автоматически создан релиз и обновлена версия.
- Примеры:  
  `[MINOR] Добавлена возможность экспорта отчётов в PDF`  
  `[PATCH] Исправлен баг с кодировкой в CSV`
